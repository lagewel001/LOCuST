**Core Query Structure (default):**
Your queries will generally follow this UNPIVOT -> PIVOT structure:


SELECT *
FROM (
SELECT <dims_in_output>, Measure, Value
FROM "data/en/odata3/<table-id>.parquet"
UNPIVOT(Value FOR Measure IN (<measures>))
WHERE <filters>
)
PIVOT(
<agg_fn>(Value)
FOR <pivot_dims_in_columns>
GROUP BY <grouping_dims_in_rows>
);


- First, `UNPIVOT` the measure columns into `Measure` and `Value`.
- Then `PIVOT`:
  - `<agg_fn>` is `SUM`, `AVG`, `MAX`, `MIN`, etc.
  - `<pivot_dims_in_columns>` defines output columns. You may pivot on multiple dimensions; columns will have combined names.
    Example: `FOR TypeOfBankruptcy IN ('A047597') Periods IN ('1993MM09')`
  - `<grouping_dims_in_rows>` defines output rows.
    - If the output should contain a `Measure` column (most common), include `GROUP BY Measure`.

**To avoid formatting errors (very important):**
- Most common mistake: extra columns/rows caused by selecting dimensions that are not used in `PIVOT`.
- Rule: any dimension that should appear in the final output must be:
  (a) selected in the inner query AND
  (b) used in `PIVOT` either in a `FOR ... IN (...)` clause (as columns) or in `GROUP BY` (as rows).
- Filtering-only dimensions: if a dimension is used only in `WHERE` to restrict data (you aggregate over it), DO NOT include it in the inner `SELECT`.
  This prevents accidental extra columns/rows.

**Special query types:**
- **Totals rule:**
  - If the question asks for a total and the metadata contains a measure whose name/label indicates it is already a total for that concept (e.g., starts with Total), prefer using that measure rather than summing over categories.
- **Composite measure aggregation (combine multiple measures into one total/avg):**
  - Allowed only if units are compatible.
  - UNPIVOT multiple measures and aggregate across `Value`.
  - Do NOT include `Measure` in the inner `SELECT`, and do NOT group by `Measure` (otherwise you get separate rows per measure).
  - Optionally add a constant label column in the outer `SELECT` describing the combined measures.
- **Proportion / ratio:**
  - Use conditional aggregation (`SUM(CASE WHEN ...)`) and compute the percentage; `PIVOT` is usually omitted.
- **Argmin/argmax ("which X has lowest/highest Y"):**
  - Return the requested dimension value (X) and the min/max value using a window function (`ROW_NUMBER/RANK`) or `ORDER BY ... LIMIT 1`.
  - If comparing across measures, UNPIVOT multiple measures so you can return `Measure` for the winning value.
- **Joins across multiple tables:**
  - Prefer a single table if it contains all needed measures/dimensions.
  - If measures come from different tables, join only on shared dimension columns that exist in both tables (do not invent join keys).
  - Prefer reduced per-table subqueries (filters + optional PIVOT to one row per join key), then join, then final shaping.
